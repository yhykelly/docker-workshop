## 1. Docker database as a local development tool

current understanding:

> For local development, it is fine to run PostgreSQL inside a Docker container, as long as the database data is stored in a Docker volume.

The key distinction is:

```text
Postgres container = runs the PostgreSQL server process
Docker volume      = stores the actual database files/data
```

A container can run the database software, but the data should not live only inside the container’s temporary filesystem.

Example:

```yaml
services:
  postgres:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: password
      POSTGRES_DB: mydb
    volumes:
      - pg_data_volume:/var/lib/postgresql/data

volumes:
  pg_data_volume:
```

Here:

```text
postgres container
  runs PostgreSQL

pg_data_volume
  stores PostgreSQL data files
```

Benefits for local development:

| Benefit              | Meaning                                                |
| -------------------- | ------------------------------------------------------ |
| Easy setup           | No need to install PostgreSQL directly on your machine |
| Reproducible         | Teammates can run the same database setup              |
| Disposable container | You can remove/recreate the container                  |
| Persistent data      | Data survives if stored in a named volume              |
| Good for testing     | Easy to reset or isolate environments                  |

Important considerations:

| Consideration                                                                 | Meaning                                                                    |
| ----------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| Use a volume                                                                  | Otherwise data may disappear when container is removed                     |
| Do not share one Postgres data volume between two running Postgres containers | Risk of corruption                                                         |
| Use compatible Postgres versions                                              | Reusing an old volume with a different major version may require migration |
| `docker compose down -v` removes volumes                                      | This deletes the persisted database data                                   |

Refined understanding:

> A Dockerized database is good for local development because the container runs the database server and the volume stores the durable data separately.

## 2. Running an app container with an external database

We discussed two patterns: local development and production.

### Local development

You may have:

```text
app container
   ↓ Docker network
postgres container
   ↓
Docker volume
```

The app connects to Postgres through the Docker Compose network.

Example:

```yaml
services:
  app:
    build: .
    environment:
      DATABASE_URL: postgres://postgres:password@postgres:5432/mydb
    depends_on:
      - postgres

  postgres:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: password
      POSTGRES_DB: mydb
    volumes:
      - pg_data_volume:/var/lib/postgresql/data

volumes:
  pg_data_volume:
```

Inside the app container, the database host is:

```text
postgres
```

not:

```text
localhost
```

because `postgres` is the service name in Docker Compose.

### Production

In production, the app can still be in a container, but the database is usually external and managed.

Example:

```text
Cloud Run app container
   ↓
Cloud SQL database
```

The container contains:

```text
your app code
dependencies
database client library
connection logic
```

Cloud SQL contains:

```text
tables
rows
schemas
users
backups
database storage
```

So the database is not inside the app container. The app container just connects to it.

Production-style example:

```text
Cloud Run container
   ↓
Cloud SQL
   ↓
BigQuery / Cloud Storage if needed
```

For the earlier data pipeline context:

```text
Cloud Run Job container
   ↓
Downloads JSON
   ↓
Saves raw files to Cloud Storage
   ↓
Writes operational records to Cloud SQL
   ↓
Writes analytics records to BigQuery
```

Only the job/app code goes in the container. Cloud Storage, Cloud SQL, and BigQuery are external managed services.

## 3. Local development vs production for databases

summarized:

> Docker can technically be used for production, even for a database, but then I own the operational responsibility.

If you run Postgres in Docker yourself, whether on your laptop or a cloud VM, you are responsible for:

```text
backups
restore testing
security
upgrades
patching
monitoring
availability
scaling
storage durability
CPU/memory sizing
networking
```

The difference between local machine and cloud VM:

| Local laptop Docker database | Cloud VM Docker database     |
| ---------------------------- | ---------------------------- |
| Good for development         | Possible for production      |
| Not reliable for users       | More suitable infrastructure |
| Laptop can sleep/restart     | VM can run continuously      |
| Weak production networking   | Cloud networking/firewalls   |
| Still self-managed           | Still self-managed           |

The key idea:

```text
Local Docker Postgres = good for development/testing
Production database = usually better as managed service
```

On GCP, the production-style choice is usually:

```text
Cloud SQL / AlloyDB / Firestore / Spanner
```

not:

```text
Postgres container on a VM
```

because managed databases help with durability, backups, patching, failover, monitoring, and security.

Refined production mental model:

```text
Containerized app = replaceable compute
Managed database = durable state
```

## 4. Docker network vs Docker Compose up

There is a real Docker command called:

```bash
docker network
```

Examples:

```bash
docker network ls
docker network create my_network
docker network inspect my_network
docker network rm my_network
```

If you use plain `docker run`, you may manually create and attach containers to a network:

```bash
docker network create my_network

docker run --name postgres --network my_network postgres:16

docker run --name app --network my_network my-app
```

But with Docker Compose, you usually do not need to manually run `docker network create`.

When you run:

```bash
docker compose up
```

Compose automatically:

```text
creates a default network
starts the services
puts the services on that network
lets services find each other by service name
```

So in Compose:

```yaml
services:
  app:
    depends_on:
      - postgres

  postgres:
    image: postgres:16
```

The app can connect to:

```text
postgres:5432
```

because `postgres` is the service name and Docker Compose provides internal DNS.

Simple distinction:

| Approach              | Network handling                                      |
| --------------------- | ----------------------------------------------------- |
| `docker run` manually | You may need `docker network create` and `--network`  |
| `docker compose up`   | Compose creates and manages the network automatically |
| Custom networking     | You can define `networks:` in Compose                 |

## 5. Use of Dockerfile

A Dockerfile is the recipe for building a container image.

Mental model:

```text
Dockerfile = recipe
Docker image = packaged application/environment
Docker container = running instance of that image
```

You showed this Dockerfile:

```dockerfile
FROM python:3.13.10-slim

COPY --from=ghcr.io/astral-sh/uv:latest /uv /bin/

WORKDIR /code

ENV PATH="/code/.venv/bin:$PATH"

COPY "pyproject.toml" "uv.lock" ".python-version" ./
RUN uv sync --locked

COPY ingest_data.py .

ENTRYPOINT ["python", "ingest_data.py"]
```

Meaning:

| Line                             | Purpose                                          |
| -------------------------------- | ------------------------------------------------ |
| `FROM python:3.13.10-slim`       | Start from a lightweight Python image            |
| `COPY --from=... /uv /bin/`      | Copy the `uv` package manager from another image |
| `WORKDIR /code`                  | Set working directory inside container           |
| `ENV PATH=...`                   | Use the virtual environment created by `uv`      |
| `COPY pyproject.toml uv.lock...` | Copy dependency files first for caching          |
| `RUN uv sync --locked`           | Install exact locked dependencies                |
| `COPY ingest_data.py .`          | Copy your Python script                          |
| `ENTRYPOINT ...`                 | Run the script when the container starts         |

The Dockerfile packages:

```text
Python version
dependencies
code
working directory
startup command
```

It does not store the data. Your script should write data to external systems like:

```text
Cloud Storage
Cloud SQL
BigQuery
```

For a simple data ingestion case, this Dockerfile could be used to build a Cloud Run Job container that runs `ingest_data.py`.

## 6. Port entry and localhost

This was one of the main concepts we clarified.

A port mapping like this:

```yaml
ports:
  - "5433:5432"
```

means:

```text
host/laptop port 5433 → container port 5432
```

Format:

```text
HOST_PORT:CONTAINER_PORT
```

So from your laptop, you connect to:

```text
localhost:5433
```

Docker forwards that to:

```text
postgres container:5432
```

But inside the Docker Compose network, another container connects to:

```text
postgres:5432
```

not:

```text
localhost:5433
```

because inside a container, `localhost` means “this same container.”

### Correct connection targets

| Client                                   | Connect to                                       |
| ---------------------------------------- | ------------------------------------------------ |
| App container → Postgres container       | `postgres:5432`                                  |
| Laptop DBeaver/psql → Postgres container | `localhost:5433`                                 |
| Inside Postgres container itself         | `localhost:5432`                                 |
| Browser → web app container              | `localhost:8000` or whatever host port is mapped |

Important clarification:

PostgreSQL is not a website. So if Postgres is mapped like:

```yaml
ports:
  - "5433:5432"
```

you usually do **not** open:

```text
http://localhost:5433
```

in a browser.

Instead, you use a database client:

```bash
psql -h localhost -p 5433 -U postgres -d mydb
```

or a GUI like DBeaver, pgAdmin, DataGrip, or TablePlus.

For a web app container, browser access makes sense:

```yaml
services:
  app:
    ports:
      - "8000:8000"
```

Then:

```text
http://localhost:8000
```

is meaningful because the app speaks HTTP.

Simple rule:

```text
Inside Docker Compose:
service_name:container_port

From your laptop:
localhost:host_port
```

## 7. Volume vs network vs localhost

These three concepts solve different problems.

| Concept                    | Purpose                                     |
| -------------------------- | ------------------------------------------- |
| Docker volume              | Stores persistent files/data                |
| Docker network             | Lets containers communicate                 |
| `localhost` + port mapping | Lets your laptop access a container service |

For Postgres:

```text
pg_data_volume
  stores database files

Docker network
  lets app container talk to postgres container

localhost:5433
  lets your laptop talk to postgres container
```

You do not query the Docker volume directly. PostgreSQL data files are internal database files. You query the database through the PostgreSQL server process using SQL.

So:

```text
Volume = storage
Network = container-to-container communication
Port mapping = host-to-container communication
```

## 8. Final mental model

A revision version of all current understanding:

```text
Local development:
- App can run in one Docker container.
- Postgres can run in another Docker container.
- Postgres data should be stored in a Docker volume.
- Docker Compose can start both containers together.
- Docker Compose creates a network so app can connect to postgres:5432.
- If I want to connect from my laptop using DBeaver/psql, I publish a port like 5433:5432 and use localhost:5433.

Production:
- App can still be containerized, for example on Cloud Run.
- Production database should usually be external and managed, for example Cloud SQL.
- App container connects to Cloud SQL using connection settings/secrets.
- I avoid putting important production database state only inside a container.
- Managed database services reduce the burden of backups, availability, patching, monitoring, and recovery.
```

One compact takeaway:

```text
Container = runs logic/process
Volume = persists local Docker data
Network = connects containers
Port mapping = exposes container service to laptop
Managed database = safer place for production state
```
